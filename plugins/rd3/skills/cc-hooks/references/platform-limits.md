# Platform Limitations and Feature Support

What's NOT portable and why. Workarounds for limited platforms.

## Feature Support Matrix

| Feature | Claude Code | Codex | OpenCode | Pi (pi-hooks) | Gemini CLI |
|---------|:-----------:|:-----:|:--------:|:--------------:|:----------:|
| `type: "command"` hooks | ✅ | ✅ | ✅ | ✅ | ✅ |
| `type: "prompt"` hooks | ✅ | ❌ | ❌ | ❌ | ❌ |
| `type: "http"` hooks | ✅ | ❌ | ❌ | ❌ | ❌ |
| `type: "agent"` hooks | ✅ | ❌ | ❌ | ❌ | ❌ |
| Regex matchers | ✅ | ❌ | ✅ | ✅ | ❌ |
| `if` conditions | ❌ | ❌ | ❌ | ✅ | ❌ |
| Tool input modification (`updatedInput`) | ✅ | ❌ | ❌ | ✅ | ❌ |
| `additionalContext` injection | ✅ | ❌ | ❌ | ✅ | ❌ |
| Stop continuation (block stopping) | ✅ | ❌ | ❌ | ✅ | ❌ |
| SubagentStop | ✅ | ❌ | ❌ | ❌ | ❌ |
| Compact hooks | ✅ | ❌ | ❌ | ✅ | ❌ |
| Notification hooks | ✅ | ❌ | ❌ | ❌ | ❌ |
| Synchronous blocking | N/A | N/A | N/A | N/A | ✅ |

## Platform-Specific Details

### Claude Code

**Strengths:** Most comprehensive hook system. 11 event types, 4 hook types (command, prompt, http, agent), regex matchers, full tool input/output access.

**Limitations:**
- Prompt hooks require a lightweight LLM call (adds latency, ~1-3s per hook)
- Agent hooks can spawn sub-agents (expensive, use sparingly)
- Hooks execute in plugin scope — `CLAUDE_PLUGIN_ROOT` must be used for file paths

**Config location:** `.claude/settings.json` (project) or `~/.claude/settings.json` (global)

**Documentation:** https://docs.anthropic.com/en/docs/claude-code/hooks

---

### Codex

**Status:** Experimental (as of 2026-04). PreToolUse/PostToolUse recently added.

**Strengths:** Rust-based engine, concurrent hook execution, fast deterministic checks.

**Limitations:**
- Only 3 events: `session_start`, `pre_tool_use`, `post_tool_use`
- No Stop/SessionEnd hooks
- No prompt hooks (command only)
- Exact string matching only (no regex matchers)
- Focused on Bash tools initially; expanding to other tools
- No `updatedInput` or `additionalContext` support yet

**Config location:** `codex.json` (hooks section)

**Workarounds:**
- For Stop-like behavior: use a post_tool_use hook on the last expected tool
- For regex matchers: use pipe-separated exact names (`bash|grep|find`)

---

### OpenCode

**Architecture:** Plugin-based. Maps Claude Code events to OpenCode's event system via a compatibility bridge.

**Strengths:** Can read `.claude/settings.json` directly (if using the bridge plugin), TypeScript plugin API.

**Limitations:**
- No prompt hooks (command only)
- Plugin reliability varies by OpenCode version
- Native hook support is a feature request, not built-in
- Event mapping may lag behind Claude Code updates

**Config location:** `.opencode/plugins/cc-hooks.ts` (generated by our emitter)

**Workaround:** Our emitter generates a TypeScript plugin that reads `.claude/settings.json` and maps events. This means the Claude Code config must also be generated.

---

### Pi (via @hsingjui/pi-hooks)

**Strengths:** 9 events, `if` conditions (`ToolName(pattern)` syntax), tool input modification, Stop continuation, compact hooks. Closest to Claude Code in capability.

**Limitations:**
- Only `type: "command"` hooks (no prompt/http/agent)
- Requires explicit extension installation: `pi install npm:@hsingjui/pi-hooks`
- Extension is at version 0.0.1 — may change
- Tool names are lowercase (`bash` not `Bash`)
- `if` conditions only work for tool events (PreToolUse, PostToolUse, PostToolUseFailure)

**Config location:** `.pi/settings.json` (project) or `~/.pi/agent/settings.json` (global)

**Installation:**
```bash
pi install npm:@hsingjui/pi-hooks
# Then run /reload in Pi to activate
```

**`if` condition syntax:**
```json
{
  "if": "Bash(git push*)",
  "type": "command",
  "command": "echo 'git push detected'"
}
```

---

### OpenClaw

**Status:** Based on Pi. If OpenClaw supports Pi extensions, the same `pi-hooks` extension applies.

**Verification needed:** Confirm during implementation whether OpenClaw loads `.pi/settings.json` and supports the pi-hooks extension.

**Limitations:** Same as Pi, plus potential OpenClaw-specific constraints.

---

### Gemini CLI

**Architecture:** Synchronous middleware. The agent loop pauses entirely while a hook executes.

**Strengths:** Deterministic execution order, good for governance/compliance.

**Limitations:**
- Only 3 events: `BeforeTool`, `AfterTool`, `AfterAgent`
- No session start/end hooks
- No prompt hooks (command only)
- Exact string matching (no regex)
- Synchronous blocking — hooks must complete before the agent continues
- Communication via stdin/stdout JSON (hook commands must handle this protocol)
- No tool input modification or additionalContext injection

**Config location:** `.gemini/settings.json`

**Stdin/stdout protocol:**
- Hook receives JSON on stdin: `{"tool_name": "bash", "tool_input": {...}}`
- Hook must output JSON on stdout: `{"decision": "allow"}` or `{"decision": "deny", "reason": "..."}`
- Non-JSON output causes a parsing failure

**Documentation:** https://geminicli.com/docs/hooks/reference

---

### Antigravity

**Status:** No lifecycle hooks. Uses Workflows triggered by manual slash commands.

**What works:** Manual `/plan`, `/review`, `/test` commands that execute predefined workflows.

**What doesn't work:** Automatic hook execution on tool calls, session events, or stop events.

**Recommendation:** Document workflow equivalents in your project's CONTRIBUTING.md. No config generation possible.

## Workarounds Summary

| Need | Claude Code | Codex/OpenCode | Pi | Gemini |
|------|-------------|----------------|-----|--------|
| Prompt-style validation | Native `type: "prompt"` | Command hook + LLM CLI | Command hook + LLM CLI | Command hook + stdin/stdout |
| Regex matchers | Native | Pipe-separated exact names | Native regex | Exact match only |
| Stop continuation | Native | N/A | Native (pi-hooks) | N/A |
| Tool input modification | Native `updatedInput` | N/A | Native (pi-hooks) | N/A |
| Context injection | Native `additionalContext` | N/A | Native (pi-hooks) | N/A |

## Recommendations

1. **Design for the lowest common denominator:** Use `type: "command"` hooks with pipe-separated tool names. This works on all 5 platforms.

2. **Layer platform-specific features:** Add prompt hooks, `if` conditions, or regex matchers as platform-specific enhancements in the abstract config. Emitters handle the filtering.

3. **Test on each target platform:** Don't assume a hook that works on Claude Code works identically on Pi. The execution model differs (sync vs async, stdin/stdout protocol, etc.).

4. **Pin pi-hooks version:** The `@hsingjui/pi-hooks` extension is at 0.0.1. Pin to a known-good version in your project's docs.
