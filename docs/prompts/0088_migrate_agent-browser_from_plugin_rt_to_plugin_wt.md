## 0088. migrate agent-browser from plugin rt to plugin wt

### Background

Despite we have a Agent Skill and a subagent which both of them are named agent-browser, we need to migrate them to plugin wt with new relevant standards. You can refer to the existing stuffs from:

- plugins/rd/skills/agent-browser
- plugins/rd/agents/agent-browser.md

### Requirements

- Use slash command `/rd2:skill-add wt markitdown-browser` to create a new Agent Skills. To avoid confusing, we use another name `markitdown-browser` instead of the original one.

- Use slash command `/rd2:agent-add wt agent-browser` to create a new subagent into plugin wt.

- Extract useful information from the above two information sources, and update the new Agent Skill and subagent with relevant information.

- Use slash command `/rd2:skill-evaluate plugins/wt/skills/markitdown-browser` and `/rd2:skill-refine plugins/wt/skills/markitdown-browser` to evaluate and refine the new Agent Skill.

- Use slash command `/rd2:agent-evaluate plugins/wt/agents/agent-browser.md` and `/rd2:agent-refine plugins/wt/agents/agent-browser.md` to evaluate and refine the new subagent.

### Solutions / Goals

**Technology Stack:**
- agent-browser CLI for browser automation
- MarkItDown CLI for document conversion
- wt plugin structure

**Implementation Approach:**
1. Create new Agent Skills directory: `plugins/wt/skills/markitdown-browser/`
2. Create new subagent file: `plugins/wt/agents/agent-browser.md`
3. Extract and migrate relevant information from existing sources
4. Update with new standards and combined functionality

#### Q&A

**Q:** Why rename the skill to `markitdown-browser`?
**A:** To avoid confusion with the existing `agent-browser` skill in plugin rd. The new name reflects the combined functionality of browser automation (agent-browser) and document conversion (MarkItDown).

**Q:** What information was extracted from the existing sources?
**A:** Core browser automation commands, snapshot-first workflow, ref-based interaction patterns, MarkItDown commands, and best practices for both tools.
